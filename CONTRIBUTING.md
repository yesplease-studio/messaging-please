# Contributing

messaging-please is an opinionated framework. Contributions that sharpen the methodology, add real-world grounding, and improve the quality of messaging deliverables are welcome. Contributions that add complexity or sand the edges off the framework's point of view are not.

## What's worth contributing

- **New principles** grounded in real cases -- with the case attached. Don't add a principle without evidence.
- **Improvements to skill workflows** based on what broke or was awkward in practice. Describe what happened and what the fix addresses.
- **New examples** that show the framework applied to a real (or plausibly real) messaging situation.
- **Corrections to the banned phrases list** -- additions or removals, with the reasoning.
- **Interoperability improvements** -- better handoff specs between messaging-please and its siblings, grounded in real usage patterns.

## What isn't a contribution

- Adding caveats or disclaimers to the positioning framework
- Making the principles more neutral or less opinionated
- Duplicating methodology that belongs in a sibling (sales qualification in sales-please, voice conventions in voice-please)
- Adding more skills without a clear gap the existing skills don't cover

## How to contribute

1. Open an issue describing the change and the reason before writing prose or code.
2. Reference a real case if possible. The framework is stronger when principles trace to actual companies.
3. Keep changes minimal. A targeted edit to one skill or one principle is better than a broad refactor.
4. Submit a PR with a clear description of what changed and why.

## Formatting

Follow the voice and format conventions in the existing SKILL.md and SYSTEM.md files. Direct, specific, practitioner register. No hedging, no filler, no "it depends" without saying what it depends on.
